⚡ 阶段一：红队安全审计与 API 绕过漏洞挖掘报告

红队在纯只读静态代码和 API 路由审计阶段，发现多处能够绕过前端界面、直接通过原始 API 对系统质量门禁构成严重威胁、或污染测试资产数据库的致命业务逻辑漏洞：

严重级别 攻击点 / 漏洞点 PRD 依据 代码/逻辑证据 复现路径与潜在风险
CRITICAL 门禁绕过 (Document-Gate Bypass) FR-REQ-009 / FR-CASE-001 POST /api/cases/ 缺少对文档 stored 校验。 调用 `create_case` 时如果强行传入未审核入库、甚至处于 `check_failed` 状态的 `source_document_id` 依然可以顺利建正式用例，完全架空需求合规门禁。
CRITICAL 高可信篡改 (Trust Status Tampering) Global Guardrails / 人工确认 TestCaseUpdate 直接对外暴露 `trust_status` 与 `is_active`。 攻击者可直接发送 PUT 请求将一个没有任何执行和验证基础的新用例篡改为 `verified` 或 `high_value`（高价值），制造质量虚假繁荣。
HIGH 执行语义篡改 (Semantic Manipulation) FR-CASE-005 / 安全阻断 CaseService.update_case 关键属性变更没有降级重设。 攻击者若手动修改正式 case / 自动化 case 的 steps、precondition、expected_result 核心数据，原有的 verified 可信度不仅不归零/降级，甚至 `pending_reconfirm` 不会置1，导致带病用例被直接触发自动化执行。
HIGH 垃圾空白数据污染 (Orphan/Ghost Case) FR-CASE-001 / 数据质量 API 层面没有字段完整性保护。 攻击者直接提交 steps, precondition, expected_result 等全是空白的“幽灵用例”依然可以通过校验并入库，导致大量的垃圾空数据垃圾污染。
MEDIUM 负样本收集沉淀缺位 (Feedback Sinking Defect) FR-CASE-007 / 负向语料 drafts/reject 只是简单置为 rejected。 当人工拒绝 AI 生成的高风险/幻觉草稿时，没有任何结构化原因和详细改进意见（字数门槛限制），负样本库无法反哺 AI，从而导致 AI 多次重复犯同样的严重错误。
MEDIUM 待办中心模块彻底缺失 (Todo Center Absence) FR-TODO-001 / FR-TODO-008 整个 Todo 模块属于空白未实现状态。 需求解析失败、待复核、待确认等事件无法触发待办事件注册，测试和产品无法协同，严重影响人机交互效率。

⚡ 闭环负反馈消费黑科技：RagRetrieveSkill 负向 Few-Shot 注入机制

除了高相似度门禁拦截外，蓝队还在 AI 用例生成的核心引擎（CaseGenerateSkill）里，实现了自适应的少样本负向强化学习（Few-Shot Negative Learning）：

1. 动态语义语料召回

当 AI 生成测试用例时，系统会自动将当前文档的 title 结合解析的 requirement 汇总为 query，使用自研的 RAG 检索引擎（`RagRetrieveSkill`）到 `negative_case_samples` 负样本数据库中查询相似度高、在历史中曾被人工退回的典型错误用例（最多 3 条）。

2. 负向约束动态注入 Prompt

查询召回历史负样本语料后，系统会全自动将其格式化为 “过去被拒绝的错误用例模式（请务必避免）”，并动态拼接在 LLM 的最终设计 Prompt 中作为极强的反向规避和特征约束，从而让 AI 拥有“汲取历史教训”的能力，杜绝幻觉和质量退化。
