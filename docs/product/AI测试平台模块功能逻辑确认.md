# AI 测试平台模块功能逻辑确认

版本：v2.0  
文档角色：模块边界确认与 PRD 索引  
主文档：`docs/product/AI测试平台PRD.md`  
说明：完整产品需求、用户故事、功能需求、验收标准、护栏、数据 / API / UI 影响和 AI 评估策略均以 PRD 为准。本文档不再重复承载需求正文，避免同一功能在多个文档中出现不同口径。

## 1. 本轮确认结论

- `docs/product/AI测试平台PRD.md` 是唯一产品需求主文档。
- 本文档只保留模块边界、模块索引、已解决冲突和待确认事项。
- 一级模块确认为 11 个：首页 / 质量驾驶舱、需求中心、Case 中心、接口中心、移动端执行中心、执行中心、报告中心、AI 能力中心、变更中心、待办中心、系统设置。
- 变更中心是一级模块，承接变更记录、影响分析、回归推荐和人工确认闭环。
- 待办中心是一级模块，承接 AI case、失败报告、元素失效、flaky case、变更影响、设备异常等人工复核事项。
- Bug 中心写入 PRD 作为未来完整模块；当前不开发独立页面、不做外部缺陷平台同步、不要求业务模块保存完整 bug 字段。
- 基础权限当前只在系统设置中保留配置占位，不要求所有业务模块完成真实鉴权。
- AI 输出影响正式资产时，必须通过评估和人工确认。
- 凡需求解析、接口解析、case 生成、门禁检查、覆盖评估、失败分析或其他逻辑触发 AI 调用时，不允许使用降级逻辑替代真实 AI 调用；AI 连接失败、连接错误、鉴权失败、模型不可用、超时或配置错误时，必须直接弹窗报错并阻断任务。
- 检查未通过的需求文档或接口文档不得生成 case 草稿。

## 3. 核心链路确认

### 3.1 需求到 case

PRD 依据：`## 17.1 需求到 case`。

确认条件：

- 需求中心完成上传、解析、历史对比、耦合检查和人工确认。
- 检查未通过的文档不能生成 case 草稿。
- AI 能力中心生成可评估的 case 草稿。
- Case 中心完成 AI 草稿确认、拒绝、合并和正式入库。
- AI 草稿被拒绝后，系统强制采集结构化拒绝分类与详细原因，并自动转化为安全脱敏的负样本存入 RAG 向量数据库，在后续生成中自动注入负向少样本约束（Few-Shot Negative Learning）并触发门禁拦截。
- 正式 case 被废弃时，必须强制采集废弃原因并将其逻辑标记隔离（`is_active=0`），流转至已归档视图，执行中心任何运行场景必须 100% 过滤已废弃的用例。
- 正式 case 必须能追溯需求点、原文片段、AI 原始输出和确认人。

### 3.2 接口 case 到报告

PRD 依据：`## 17.2 接口 case 到报告`。

确认条件：

- 接口中心提供已确认接口定义。
- Case 中心保存接口 case 步骤、变量提取和断言。
- 执行中心统一创建任务、执行、重试、取消和失败分类。
- 报告中心保存 request、response、assertion、log 和 HTML 报告。

### 3.3 Android 移动端执行到报告

PRD 依据：`## 17.3 Android 移动端执行到报告`。

确认条件：

- 移动端执行中心识别 Android 设备并管理设备锁。
- App package、version 和 build 绑定到执行任务 and 报告。
- 元素资产绑定页面、模块、App 版本或页面版本。
- 正式移动端 case 必须引用元素库。
- 报告中心保存截图、XML、Appium 日志和设备异常证据。
- **定位失效自愈机制**：Appium 运行定位失败时自动抓取 UI 截图、DOM XML 树及 Appium 通信日志；AI 自动匹配并计算出置信评分 >= 0.70 的全新定位推荐方案，推送至待办中心；用户点击应用方案后自动改写对应 `MobileElement` 并核销待办。

### 3.4 变更到回归推荐

PRD 依据：`## 17.4 变更到回归推荐`。

确认条件：

- 变更中心记录需求、接口、元素、App、case、bug 修复或报告趋势变化。
- **依赖拓扑追踪算法**：变更发生时，自动在后台沿“需求链”、“接口链”、“元素链”进行最大深度不超过 5 层的依赖图谱递归分析。同时结合 RAG 进行语义相关性推荐。
- 变更中心输出受损用例清单、回归推荐集和 AI 变更自适应修改草稿。
- AI 修改草稿作为影子分支/草稿保存，合并确认前严禁篡改正式资产。
- 待办中心承接人工复核，人工确认前不得修改正式资产。

### 3.5 待办核销与生命周期闭环

PRD 依据：`## 14. Module: 待办中心`。

确认条件：

- 待办中心作为统一方工作入口，集中监听和流转来自 AI 待确认草稿、失败复核、元素失效、接口变更等 8 种事件。
- **自清洗/自动核销闭环 (Self-Cleaning)**：彻底避免重复处理。当用户在对应业务模块中（如 Case 中心确认/拒绝草稿，或移动中心应用自愈推荐）完成最终决策后，待办中心必须 100% 自动将该 TODO 标记为 `resolved` 归档。
- **运行通过核销**：失败用例在被重新运行通过后，关联的失败复核 TODO 必须由系统自动核销。
- SSE/WebSocket 实时向前台推送待办计数气泡，手动忽略/挂起强制校验 10 字以上理由。

## 4. 已解决的边界冲突

| 冲突点 | 最终结论 | PRD 落点 |
|---|---|---|
| 变更中心是否一级模块 | 是，作为一级模块纳入 PRD。 | `## 13. Module: 变更中心` |
| 待办人工复核队列是否一级模块 | 是，作为一级模块纳入 PRD。 | `## 14. Module: 待办中心` |
| Bug 是否当前模块 | 不是当前开发模块，是未来完整模块。 | `## 16. Future Module: Bug 中心` |
| 基础权限是否当前完整鉴权 | 不要求完整鉴权，只在系统设置保留配置占位。 | `FR-SET-010` |
| AI 能力治理是否独立模块 | 不独立，全部归入 AI 能力中心。 | `## 12. Module: AI 能力中心` |
| 用例资产治理 / 回归治理是否独立模块 | 不独立，全部归入 Case 中心。 | `## 7. Module: Case 中心` |
| AI 调用失败是否允许降级 | 不允许。AI 连接失败或连接错误必须弹窗报错并阻断任务，不得生成降级结果。 | `FR-AI-013`、`FR-SET-012` |

## 5. 后续待确认事项

以下问题不影响当前 PRD 作为主文档使用，但会影响原型、技术设计或验收细化：

- 需求树页面如何最直观表达业务域 -> 模块 -> 功能 -> 需求版本 -> 验收点。
- 解析失败时是否允许手动粘贴文本作为兜底输入。
- Prompt 修改验证使用完整评估集还是轻量冒烟集。
- Evaluator 阈值的模块和风险等级枚举如何定义。
- 待办状态是否需要加入 assignee、due date 和 SLA。
- 变更类型枚举和默认状态流转如何定义。
- 报告默认保留 14 天是否符合团队实际需要。
- 基础权限占位字段是否提前设计为角色、用户、项目权限三层。

## 6. 维护规则

- 修改产品需求时，优先修改 `docs/product/AI测试平台PRD.md`。
- 本文档只同步模块边界、索引和确认结论，不复制完整需求正文。
- 如果 PRD 和本文档冲突，以 PRD 为准，并更新本文档的索引或确认结论。
- 如果代码实现和 PRD 冲突，先按 PRD 做差异清单，再决定是修代码还是更新 PRD。
