# AI 测试平台技术架构

**状态：** 现行
**产品依据：** [AI 测试平台 PRD](../product/AI测试平台PRD.md)
**模块边界：** [模块功能逻辑确认](../product/AI测试平台模块功能逻辑确认.md)

## 文档边界

本文只描述实现结构和技术约束。产品目标、功能范围、验收标准和非目标均以 PRD 为准；本文不得成为第二份需求事实源。

## 总体结构

```text
Vue 前端
  -> FastAPI API / DTO
  -> 领域 Service
  -> Harness
  -> Runner
  -> Report Service

领域 Service
  -> AI Skill / Prompt / Evaluator / RAG
  -> Repository / Storage
```

- API 层只做协议、校验、依赖和兼容转发。
- Service 编排业务流程、状态和事务。
- Harness 准备环境、变量、依赖、测试数据、清理和证据上下文。
- Runner 只执行协议动作并返回标准结果，不写需求、Case 或报告主表。
- Skill 只封装单一 AI 能力；Evaluator 只给门禁结论，均不得替代人工确认。
- Report Service 根据执行结果和产物生成可复盘证据。

## 业务域

后端目标业务域为需求、Case、接口、执行、报告、移动端、变更和待办；AI 与平台底座作为跨域能力。前端模块与这些业务域保持稳定映射。

当前目录仍保留部分横向组织，后续目录迁移必须使用兼容层逐批完成。本批次不移动代码，也不改变 API、数据库 schema 或业务行为。

## 数据与存储

- 关系型数据保存需求、Case、接口、执行、报告、AI 调用和治理状态。
- 原始上传、日志、SQLite、截图、XML、导出报告和构建缓存属于本地运行产物，不进入 Git。
- 正式测试 fixture 必须放在受版本控制的测试 fixture 目录，不能混入真实上传目录。
- AI 输入、RAG 内容、日志和报告导出必须先经过脱敏。

## 关键调用约束

```text
API Router -> Service -> Harness -> Runner -> Report Service
```

1. Router 不直连 Runner。
2. Runner 不修改正式业务资产。
3. AI 输出不直接写正式需求、Case、质量结论或变更结果。
4. 所有正式 AI 资产经过 Evaluator 和人工确认。
5. 需求差异、覆盖矩阵和生成计划分别保持结构化事实，不以自然语言摘要替代。
6. 兼容路径只转发，不复制业务逻辑。
