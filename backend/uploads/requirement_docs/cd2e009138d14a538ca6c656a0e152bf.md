# 计算器应用高保真 PRD

版本：v2.0  
文档状态：正式版  
适用对象：前端研发工程师、测试开发工程师、系统架构师  
最后更新：2026-05-25  

---

## 1. Problem Statement

在日常移动办公与财务计算场景中，用户频繁需要一个具备高响应性、输入高容错性、且能快捷复用历史记录的轻量计算器。目前的简易计算器往往在极值运算、高精度浮点数处理以及多步连续运算的业务逻辑设计上不完善，导致用户在进行精细数据核算时难以获得连贯、可信的计算体验。

本 PRD 旨在为计算器应用的核心算法与界面设计提供规范标准，确保应用在具备极限容错能力的同时，展现逻辑严密的高精度四则运算和历史记录回填体验。

---

## 2. Goals

- **核心运算精度**：系统必须支持基于高精度十进制标准（Decimal）的浮点数运算，最大支持 15 位有效数字，彻底修正经典浮点数计算误差（如 `0.1 + 0.2` 面板必须展现为正确的 `0.3`）。
- **极限容错能力**：系统对除以 0、超长数字溢出、非法连续点击运算符等边界场景必须具备完善的阻断机制，并在界面上提供清晰、直观的视觉反馈。
- **计算历史复用**：系统支持本地持久化保存最近 20 条成功计算历史，用户可一键点击复用历史表达式，提升多步骤数据核算的连贯效率。

---

## 3. Non-goals

- 本系统不支持任何三角函数（sin, cos, tan）、对数、指数等科学计算器功能。
- 本系统不支持多币种汇率实时换算、物理单位转换以及复杂图形化函数绘制。
- 本系统不要求提供云端多端同步能力，所有历史记录均持久化在浏览器本地存储（Local Storage）中。
- 本系统不支持括号嵌套、变量定义或自定义公式等高级脚本解析器功能。

---

## 4. Personas / Users

- **财务助理（小张）**：需要频繁进行折扣、税率折算以及基础加减乘除计算，对运算精度的要求极高，绝不能出现浮点数误差，且经常需要翻看前几笔计算的历史记录。
- **中小学生（小明）**：用来检验课后作业的四则运算结果，希望输入时在按错键时能有快速退格修正功能。
- **测试开发工程师（李工）**：依据本 PRD 提炼核心功能点（FR），编写对应的功能测试用例与自动化测试脚本，执行覆盖率评估与质量门禁核销。

---

## 5. User Stories

1. 作为财务助理，我想输入带有多位小数的数值进行加减乘除，以便确保计算出的资金账目 100% 精确。
2. 作为财务助理，我想在点击百分比键时自动将当前数字折算，以便快速处理税率或折扣（例如：输入 `200` 再点击 `*` 再输入 `10` 并点击 `%`，应能直接得到 `20`）。
3. 作为普通用户，我想在一键清空表达式时有明确的 `AC` 指引，以便立刻开始下一轮全新的计算。
4. 作为中小学生，我想在手滑输入错误时，点击退格键仅删去最后输入的一位字符，而不是全盘推翻重来。
5. 作为普通用户，我想在计算完成后，能直接点击运算符将上一次的计算结果作为下一次计算的左操作数，以便支持多步连续运算。

## 6. Functional Requirements

### 【基础输入模块】

- `FR-CALC-001`：系统必须提供数字按键 `0-9`。用户点击后，当前数值输入区应右侧追加对应数字。
- `FR-CALC-002`：系统必须提供小数点按键 `.`。用户对同一个数字输入时，最多只能包含一个有效小数点，重复输入小数点系统应直接忽略。
- `FR-CALC-003`：当主显示区处于默认状态或刚刚完成 `AC` 清除时，页面必须默认展示单字符 `0`。
- `FR-CALC-004`：系统单次允许输入的最大有效数字位数限制为 15 位。如果用户输入第 16 位数字，系统应直接阻止输入，并触发短震动反馈，同时在页面顶端醒目提示“数字长度已达上限”。
- `FR-CALC-005`：系统在处理用户输入时，如果当前最左侧数字为单独的 `0`，则用户继续输入数字 `1-9` 时应直接覆写该 `0`。

### 【四则运算模块】

- `FR-CALC-006`：系统必须支持基础加法运算，运算符在显示区展示为 `+`，运算规则应符合数学加法。
- `FR-CALC-007`：系统必须支持基础减法运算，运算符在显示区展示为 `-`，运算规则应符合数学减法。
- `FR-CALC-008`：系统必须支持基础乘法运算，为了界面美观，运算符在显示区必须展示为 `×`，运算规则应符合数学乘法。
- `FR-CALC-009`：系统必须支持基础除法运算，为了界面美观，运算符在显示区必须展示为 `÷`，运算规则应符合数学除法。
- `FR-CALC-010`：系统必须采用高精度十进制（Decimal）算法进行底层运算。任何涉及小数的运算在页面上展示时，必须自动截断并修正精度误差。当除法运算导致无限循环小数时，系统结果展示区保留最多 12 位有效小数数字进行输出展示。
- `FR-CALC-011`：系统必须支持等号 `=` 按键触发当前表达式的求值计算。在计算完成后，若用户再次点击等号 `=` 键，系统应自动执行连续运算，将最新的运算结果进行累送更新。
- `FR-CALC-012`：系统必须支持连续多步运算模式。当用户输入 `2 + 3 × 4 =` 时，系统将默认按自左向右顺序得出结果为 `20`。

### 【辅助操作模块】

- `FR-CALC-013`：系统必须提供 `AC`（Clear All）清空键。用户点击后，必须一键清空当前正在编辑的表达式、计算结果，并隐藏任何页面上的错误提示。
- `FR-CALC-014`：系统必须提供退格键 `⌫`（Backspace）。用户点击后，应删除当前输入数值的最后一位字符；若当前数值已被完全删空，则显示区应立刻复原为默认的 `0`。
- `FR-CALC-015`：系统必须提供正负号切换按键 `+/-`。点击后，应使当前操作数在正数和负数之间取反切换（如 `5` 变为 `-5`，`-2.5` 变为 `2.5`）。
- `FR-CALC-016`：系统百分比运算支持复合表达式计算。当用户点击百分比键 `%` 时，系统立刻将当前数值折算为原数值的百分之一（如输入 `50` 并点击 `%`，显示区转换为 `0.5`）。系统应支持百分比在混合四则运算中直接参与计算。
- `FR-CALC-017`：系统必须具备运算符智能覆写逻辑。当用户在输入数字后，连续点击多个不同的运算符时，系统应自动以最后一次点击的运算符为准，防止生成格式错误的表达式。

### 【历史记录模块】

- `FR-CALC-018`：系统必须支持保存最近 20 条成功计算的历史记录。每条记录必须包含完整的计算表达式、计算结果以及计算时间，以便用户查阅。
- `FR-CALC-019`：当成功计算的历史记录总数达到 20 条上限后，用户若产生第 21 条计算，系统应自动执行循环覆盖，删除最早的一条记录，确保本地存储不超限。
- `FR-CALC-020`：历史记录区域支持快捷回填功能。当用户点击历史面板中的任意一条已保存的记录时，系统把该记录的数据重新填充到主显示区，用户可以直接在当前面板中继续追加后续的运算步骤。
- `FR-CALC-021`：系统必须提供一键清空历史记录功能。点击“清空历史”按钮时，系统应弹出二次确认弹窗。用户确认后，本地存储的历史记录必须被彻底安全删除。

### 【异常处理模块】

- `FR-CALC-022`：当用户在计算时执行除以 `0` 并点击等号 `=` 时，系统不得发生闪退或崩溃，应直接阻断计算，并在结果显示区高亮展示错误提示“不能除以 0”。
- `FR-CALC-023`：当用户输入的表达式不完整（例如以运算符结尾 `8 +`，或直接点击等号 `=`）时，系统应直接阻断求值，并提示“表达式不完整”，同时保持原表达式不被清除，允许用户继续补充输入。
- `FR-CALC-024`：若计算结果的值超过 15 位数字或出现极限浮点数溢出，系统应自动将展示格式切换为科学计数法（例如 `1.23456789012345e+15`）。
- `FR-CALC-025`：在系统底层检测到未知异常时，应进行全局捕获，展示通用容错页面，并写入本地错误日志。

### 【键盘支持与可访问性模块】

- `FR-CALC-026`：系统页面必须支持全键盘操作。键盘上的数字键 `0-9`、小数点 `.`、运算符 `+,-,*,/`、回车键 `Enter`（对应等号）、退格键 `Backspace` 以及清除键 `Esc` 应与页面上的虚拟按键行为保持一致。
- `FR-CALC-027`：系统对任何按键的响应、屏幕渲染及结果更新时间必须控制在 50ms 以内，确保视觉交互流畅。
- `FR-CALC-028`：对于残障用户，页面所有虚拟按键必须绑定标准的 `aria-label` 属性，并支持读屏软件正确朗读出按键名称。

## 7. Acceptance Criteria

### AC-001：基础十进制高精度加法
- **关联需求**：`FR-CALC-001`、`FR-CALC-006`、`FR-CALC-010`
- **Given**：用户打开计算器且主显示区展示为默认的 `0`
- **When**：用户依次点击按键 `0`、`.`、`1`、`+`、`0`、`.`、`2`、`=`
- **Then**：系统主显示区应无延迟地展现计算结果为 `0.3`
- **And**：该笔计算应被作为第 1 条成功历史记录安全写入历史记录区

### AC-002：输入超限拦截机制
- **关联需求**：`FR-CALC-004`
- **Given**：用户已经连续点击输入了 15 位数字（例如 `999999999999999`）
- **When**：用户尝试继续点击数字键 `8`
- **Then**：系统输入区应维持原 15 位数字不变（不追加数字 `8`）
- **And**：手机或模拟设备应触发短震动反馈
- **And**：页面顶部应滑出警告提示“数字长度已达上限”，且该提示在 2 秒后自动消失

### AC-003：除以 0 阻断防护
- **关联需求**：`FR-CALC-009`、`FR-CALC-022`
- **Given**：用户已输入表达式 `100 ÷ 0`
- **When**：用户点击等号按键 `=`
- **Then**：系统不得崩溃，显示区不得出现 `Infinity` 或 `NaN` 字符
- **And**：结果展示区必须以红色字体展示错误提示“不能除以 0”
- **And**：历史记录区不得写入该笔计算

### AC-004：运算符智能覆写
- **关联需求**：`FR-CALC-017`
- **Given**：用户已输入数字 `8` 并点击了运算符 `+`
- **When**：用户在未输入第二操作数之前，立刻又点击了运算符 `×`
- **Then**：主显示区的表达式应自动更新为 `8 ×`
- **When**：用户接着输入数字 `3` 并点击等号键 `=`
- **Then**：系统应输出计算结果 `24`

### AC-005：历史记录循环覆盖与清除
- **关联需求**：`FR-CALC-018`、`FR-CALC-019`、`FR-CALC-021`
- **Given**：用户本地已持久化保存了 20 条成功计算历史记录（第 1 条至第 20 条）
- **When**：用户完成了第 21 次成功运算
- **Then**：历史记录区应仅展示最近的 20 条记录（第 2 条至第 21 条）
- **And**：原第 1 条记录应从本地存储和界面中删除
- **When**：用户点击“清空历史”按钮并点击弹出窗中的“确认”
- **Then**：本地存储和界面中的所有计算历史必须彻底抹除

## 8. Guardrails

1. **零崩溃护栏**：计算器在面对任何非法字符流、极端数字溢出或未知异常（如 `FR-CALC-025`）时，必须通过全局边界保护，保持应用处于可用状态，**禁止发生页面白屏、崩溃、或计算按钮完全锁死失效的情况**。
2. **零绕过安全护栏**：**所有涉及删除历史记录的操作（如 `FR-CALC-021`）必须强制通过二次弹窗确认，绝对不允许存在任何无需用户确认便直接静默删除历史的逻辑旁路**。

## 9. Data / API / UI Implications

### Data Implications
- **本地持久化设计**：历史记录模型应设计为 `CalculatorHistory` 实体，字段包含：`id` (UUID), `expression` (String), `result` (String), `calculated_at` (Timestamp)。
- **非持久化状态**：当前的临时输入缓存（当前操作数、当前运算符）应全部保存在组件的单向状态机中，组件销毁时可不予保留。

### API Implications
- **本轮开发不涉及后端接口**，所有计算、高精度修正、历史记录维护和键盘映射均在前端组件内闭环完成。
- 未来若支持云同步，需扩展历史同步 API：`POST /api/v1/calculator/sync`。

### UI Implications
- **界面栅格化布局**：计算器按键面板应采用标准的 4x5 栅格化流式布局，顶部为主显示区，紧随其下为结果/错误提示区，侧边或底部为可伸缩的计算历史记录侧边栏。
- **色彩规范**：常规按键使用中性色，等号键 `=` 和错误提示（如“不能除以 0”）必须使用鲜艳醒目的橙色或红色进行强调。
