# QA 测试用例技能

当用户要求生成、补全或评审测试用例时，按以下模板与规范输出。

本 Skill 与运行时 `backend/app/skills/case_generate_skill.py::CaseGenerateSkill` 使用同一套团队 QA case 契约。聊天输出使用 Markdown 表格；平台运行时使用同义 JSON 字段，字段映射见下方“统一字段契约”。如两处存在差异，以 `CaseGenerateSkill.case_format_contract()` 为最终落地契约，并同步更新本文档。

## 执行前

- **输入范围**：用户提供的 PRD/需求摘要、页面流程、接口说明、交互原型、已有测试用例草稿，或指定的功能模块。若用户只说「写测试用例」「生成 case」等未提供任何上下文，先询问：功能/模块名称、对应需求 ID、测试目标（功能/异常/边界/回归）、输出格式（表格或列表），再生成。
- 确认至少有**功能主题 + 需求或流程信息**后，再按下方模板输出；缺项用 `[请补充：xxx]` 占位。

## 输出模式

根据用户意图，选择以下模式之一：

1. **生成模式**：从需求/流程直接生成一组测试用例。
2. **补全模式**：对已有用例补充缺失字段、异常场景、边界 case 或追溯信息。
3. **评审模式**：检查已有用例是否完整、可执行、可验证、可追溯，并输出必改 / 建议 / 可选清单。

## 用例结构（必填字段）

每条用例至少包含以下字段：

1. **用例 ID**：如 `TC-001`，若用户未指定编号规则，可用 `[待编号]`。
2. **标题**：一句话说明验证场景，建议格式为「模块 - 场景 - 预期」。
3. **关联需求**：需求 ID、用户故事 ID、接口或页面标识；无则写 `[请补充需求来源]`。
4. **前置条件**：账号状态、环境、数据、开关或依赖准备；无则写「无」。
5. **测试步骤**：按顺序编号，每步只写一个动作或一个关键验证点。
6. **预期结果**：与步骤或整条用例对应，必须可观察、可判断通过/失败。
7. **重要性**：主路径 / 高风险 / 普通 / 补充；未指定则根据用户路径、风险和影响范围判断。

## 统一字段契约

| 展示字段 | 运行时 JSON 字段 | 必填 | 说明 |
|---------|------------------|------|------|
| 用例 ID | `case_no` | 是 | 如 `TC-001`；运行时未传入时自动补齐。 |
| 标题 | `title` / `name` | 是 | 建议格式「模块 - 场景 - 预期」。 |
| 关联需求 | `requirement_ref` | 是 | 需求 ID、用户故事 ID、接口、页面或文档来源。 |
| 前置条件 | `precondition` | 是 | 账号状态、环境、数据、开关或依赖准备；无则写「无」。 |
| 测试步骤 | `steps` | 是 | 数组或编号步骤；至少 3 步，每步只写一个动作或关键验证点。 |
| 预期结果 | `expected_result` | 是 | 必须可观察、可断言。 |
| 重要性 | `importance` | 是 | 主路径 / 高风险 / 普通 / 补充。 |
| 类型 | `test_type` | 是 | 冒烟 / 功能 / 回归 / 异常 / 边界 / 兼容性 / 权限 / 接口。 |
| 测试数据 | `test_data` | 否 | 明确输入数据、样本、账号或接口参数。 |
| 备注 | `remarks` | 否 | 依赖限制、暂不覆盖说明、相关缺陷单。 |

运行时额外可保留系统字段：`category`、`priority`、`source_excerpt`、`coverage_category`、`request_data`、`request_headers`、`expected_status`、`expected_body`、`expected_contains`、`assertions`、`dependency_consideration`。

## 用例结构（可选字段）

- **测试类型**：冒烟 / 功能 / 回归 / 异常 / 边界 / 兼容性 / 权限。
- **测试数据**：明确输入数据、样本或账号。
- **备注**：依赖限制、暂不覆盖说明、相关缺陷单。
- **执行结果字段**：结果、执行人、执行时间、缺陷单号（若团队把设计与执行放在同一模板中）。

## 用例书写规范

- **标题**：避免笼统写法（如「登录测试」），改为具体场景（如「登录 - 输入正确账号密码后成功进入首页」）。
- **步骤**：必须可执行，避免「进行登录操作」这类抽象描述，改为「输入已注册邮箱与正确密码，点击登录按钮」。
- **预期结果**：必须可验证，避免「功能正常」「结果正确」，改为「跳转至首页，页面右上角显示用户名，接口返回 200 且 body 含 token」。
- **覆盖分类**：默认覆盖主路径、异常路径、边界条件；若用户只要求核心 case，可先生成主路径和高风险 case。
- **关联需求**：每条用例至少关联一个需求 ID 或来源描述，避免孤立 case。
- **避免重复**：多个 case 仅在输入或前置条件不同的，应明确区分差异，不要只换标题。

## 输出格式

- 默认输出为 **Markdown 表格**；若步骤较长或每步都需写预期，可改为**分条列表**。
- 若生成的是一组用例，先给出**覆盖范围摘要**（覆盖了哪些主路径/异常/边界），再给出 case 明细。
- 占位符统一为 `[请补充：xxx]` 或 `[待确认]`。

## 示例输出（节选）

```markdown
## 覆盖范围摘要
- 核心登录成功路径
- 密码错误异常路径
- 未注册邮箱异常路径
- 密码为空边界场景

| 用例 ID | 标题 | 关联需求 | 前置条件 | 步骤 | 预期结果 | 重要性 | 类型 |
|---------|------|----------|----------|------|----------|--------|------|
| TC-001 | 登录 - 正确账号密码登录成功 | R1 | 用户已注册且账号状态正常 | 1. 打开登录页 2. 输入已注册邮箱 3. 输入正确密码 4. 点击登录 | 跳转至首页，显示登录态，接口返回 200 且含 token | 主路径 | 功能 |
| TC-002 | 登录 - 密码错误时提示错误信息 | R1 | 用户已注册且账号状态正常 | 1. 打开登录页 2. 输入已注册邮箱 3. 输入错误密码 4. 点击登录 | 页面提示“账号或密码错误”，停留在登录页，不创建登录态 | 高风险 | 异常 |
| TC-003 | 登录 - 密码为空时按钮置灰或提示必填 | R1 | 无 | 1. 打开登录页 2. 输入已注册邮箱 3. 密码留空 4. 点击登录按钮 | 按钮不可点击或提示“密码不能为空”，不发起登录请求 | 补充 | 边界 |
```

若用户提供的是**已有测试用例**并要求「评审」「检查是否规范」，则输出：符合项 ✓、缺失字段、不可执行步骤、不可验证预期、未覆盖的关键场景；按 **必改 / 建议 / 可选** 分级，并给出修改建议。
